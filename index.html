Cycle-time inference

End-to-end data pipeline & visualization app in the context of an heavy industry for my MSc thesis

Motivation

The strategic goals in industries in heavy mature industries, with low product differentiation is through price competition.
The value proposition is to capture real time data from industrial programmable logic circuits through the shop-floor network and store it into structured databases, to compute industry relevant metrics and to build a dashboard to support managerial decisions.

Keywords: tire-industry; data-mining; data-visualization; support-vector machines;

Deliverables

  1. Data pipeline with emphasis in data validation
  2. Develop a data validation tool
  3. Create a structured database
  4. Build a data visualization dashboard

Problems

  1. Machines generations had up to 50 years age gaps
  2. Data integrity at source
  3. Time-memory tradeoff & solution complexity optimization
  4. Communication barriers between stakeholders

Remarks

  1. Agile methodologies are more suitable for these projects
  2. Spend more time planning & interacting with users, and less time coding
  3. Reusable web components and elements brings down UIX time-to-market
  4. ORM isn't optimal with simple models & high throughput

#Data-Engineering

Staging

The staging workflow pulls the data from the machine controllers database into a staging SQL cloud database with basic schema validation.
Potential for improvement:

  1. replace a scheduler for a streaming service such as Spark;
  2. add concurrency between tasks;
  3. robust validation with pandera
  • Data sources: engineering services, programmable logic circuits from production lines
  • Data transformations: group by machine id
  • Data target: MYSQL RDBMS database from company cloud infrastructure
    (retention period 3 months)
  • Schema: unstructured
  • Failure modes:
    1. power outage;
    2. shopfloor network access revogued;
    3. cloud access revogued;
    4. dask validation schema failure;
  • Scheduling: 1 hour
  • Tools: sqlalchemy, urllib, dask, sockets, pandas, airflow
      • Worflow:

      Connect machine ip >> Fetch data>> Insert data

      1. linear workflow, there is no concurrency between machines;
      2. the workflow is triggered by fetch machine ips task;
      3. there are no dependencies between machines (if one fails, others are unaffected);
      4. all-or-nothing behavior, if one partition fails, all fail;
    Data mart

    The data mart workflow structures the data into a Snowflake schema.

  • Data sources: staging database
  • Data transformations:
    1. denormalize fact table;
    2. split fact table into adjacent dimension tables (such as recipe, tire-size, shift, operator, etc) & subdimension tables (date)
    3. split dimension tables into subdimension tables (such as date, shift)
    4. handling missing data;
  • Feature-engineering:
    1. decompose date into subparts (day, week, weekday, month, etc);
    2. label encode categorical variables;
  • Data target: MYSQL RDBMS local application database
  • Schema: Snowflake Schema
  • Scheduling: 1 day
  • Tools: sqlalchemy, dask, pandas, airflow, scikit-learn
  • Potential for improvement:
    1. include new features such as humidity, temperature, thermal sensation, canteen menu;
    2. clustering machines, operators into broader classes
